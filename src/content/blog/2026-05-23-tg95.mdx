Генератор — это итератор. Просто способ его создания другой. Можно через класс, можно через yield, можно через генераторное выражение (x for x in range(10)).

А где же ленивые вычисления?

Вот здесь начинается путаница. Часто говорят: «итераторы ленивые, считают по одному элементу». Это правда на практике, но это не часть протокола.

В протоколе итератора нигде не написано, что next обязан что-то лениво вычислять. Можно написать итератор, который при создании прогружает всё в память и потом просто отдаёт по элементу:

class BadIter:

def __init__(self, n):

self.data = [x*x for x in range(n)] # посчитали всё сразу

self.i = 0

def __iter__(self):

return self

def __next__(self):

if self.i >= len(self.data):

raise StopIteration

val = self.data[self.i]

self.i += 1

return val

Это валидный итератор. Никакой ленивости. Но снаружи неотличим от ленивого — тот же for x in ....

Ленивость — это свойство реализации, а не контракта. Генераторы через yield ленивы по построению (выполнение паузится на `yield`), но это особенность их реализации, а не требование протокола итератора.

Возвращаясь к собесу

Правильный ответ — не «генератор не грузит память».

Я бы ответил так: «Итератор — это протокол, объект с iter и next. Генератор — один из способов этот протокол реализовать, через yield, по сути синтаксический сахар, чтобы не писать класс руками. Ленивость не следует ни из того, ни из другого, это свойство конкретной реализации».
